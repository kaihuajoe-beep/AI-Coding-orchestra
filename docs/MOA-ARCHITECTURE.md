# MoA 协作架构

## 目标

MoA（Mixture-of-Agents）用于降低单模型盲区，但不把“模型数量”误当成正确性。运行时将不同家族模型作为候选方案生成器和独立评审者，最终交付仍以仓库证据、自动化测试和人工批准为准。

## 执行流程

1. 读取最多 40,000 字符的仓库上下文，排除 `.env`、凭据文件、依赖和构建目录。
2. 优先从不同家族选择 2–8 个 Agent，并行生成只读候选方案。
3. 每个成功候选交给不同家族的 Agent 评审。
4. 评审输出 0–100 分、结论和证据；无法解析的评审按 50 分、需修改处理。
5. 使用评审中位数抵抗单个异常评分，再叠加证据奖励和 reject 惩罚。
6. 可选 synthesizer 根据候选和异议生成统一方案；调用失败则回退最高分候选。
7. 输出完整决策包。执行 Agent 必须在独立分支或 worktree 应用变更并运行测试。

## MCP 调用示例

```json
{
  "name": "moa_orchestrate",
  "arguments": {
    "task": "为登录接口增加限流并补齐测试",
    "workdir": "/absolute/path/to/project",
    "candidate_count": 3,
    "reviews_per_candidate": 2,
    "max_cost_rmb": 10,
    "synthesizer": "openrouter"
  }
}
```

至少需要两个已配置的不同模型家族。`synthesizer` 可省略；省略时直接采用最高分候选。

## 安全边界

- Codex 候选与评审通过 `--sandbox read-only` 运行。
- API 模型只获得经过过滤和截断的文本快照，没有文件写入工具。
- MoA 工具不提交、不推送、不创建 PR，也不自动应用代码。
- `max_cost_rmb` 基于 Provider 返回的实际用量执行；API 请求已经发出后才能得到精确费用，因此它是结果门禁，不是预付费硬限额。
- 当前文件过滤是降低意外泄露风险的基础措施，不替代仓库级数据分类和出站策略。

## 决策原则

- 多数票只是信号，证据强度优先。
- 没有证据的高分不应覆盖可复现的反例。
- 不要求评审者必须发现固定数量的问题。
- 模型宣称“测试通过”不算验证；只有执行日志和 CI 状态算验证。
